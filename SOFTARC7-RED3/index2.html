Outline the key objects for each of the Waterfall stages so that you can assess success or failure of the stage.

Requirements:

Objects:

  1. Requirements documents.
  2. Stakeholders feedback.

Success criteria:

  1. Clear, complete requirements documented.
  2. Stakeholders expectations with documented requirements.

Failura criteria

  1. Incomplete requirements documented.
  2. Stakeholders' expectations not addressed.

Design:

Objects:

  1. Design Document.
  2. Detailed Design Specifications.

Success criteria:

  1. The structure of the system.
  2. Detailed design specifies, and data flows
  3. Design is aligns with requirements.

Failura criteria

  1. Design lacks scalability and alignment with requirements.
  2. architecture isarchitecture is incomplete.
  3. Detailed design does not adequately.

Implementation:

Objects:

  1. Source Code
  2. Test Cases

Success criteria:

  1. Code standards and best practices.
  2. Unit tests pass, validating
  3. Code is maintainable, and well documented.

Failura criteria

  1. Code does not standards or best practices.
  2. Tests fail, indicating defects in the code.
  3. Code lacks documentation.

Verification:

Objects:

  1. Verification Plan
  2. Test Strategy

Success criteria:

  1. Verification plan outlines the scope, objectives, resources, and schedule for verification activities.
  2. Test strategy defines the approach, techniques, to be used for testing.

Failura criteria

  1. Verification plan lacks clarity or does not cover all necessary aspects
  2. Test strategy is not suitable for the project's requirements

Maintenance:

Objects:

  1. Change Requests.
  2. Bug Reports.
  3. Maintenance Logs.

Success criteria:

  1. Maintenance tasks are effectively managed and prioritized.
  2. Bugs and issues are promptly fixed and tested.
  3. Updates are successfully implemented without introducing new issues.

Failura criteria

  1. Maintenance tasks are poorly managed.
  2. Fixes or updates introduce new defects.
  3. Lack of documentation or communication regarding maintenance.

Include in your submission evidence to show completion of each stage.

Requirements:

Evidence:

  1. Document detailing all gathered requirements, including functional and non functional.
  2. Formal sign from stakeholders agreement and acceptance of the documented requirements.
  3. Requirements document is approved by relevant stakeholders.

Design:

Evidence:

  1. Design Document describes the overall system architecture, including major components.
  2. Detailed specifications for each component, including data structures, algorithms.
  3. Documentation of the review meeting where the design was presented and discussed.

Implementation:

Evidence:

  1. Complete and commented codebase reflecting the implementation.
  2. Test cases designed to validate and functionality.
  3. Documentation of code review sessions and identified issues.

Verification:

Evidence:

  1. Documentation of verification activities, including test plans, test results, and any issues or defects.
  2. Records of identified defects or issues.
  3. Verification reports provide a clear overview.

Maintenance:

Evidence:

  1. Documentation of requested changes or software.
  2. Records of reported bugs or issues.
  3. Documentation of maintenance activities performed and outcomes.

In your opinion, would the Web project completion process be more, or less, difficult if a wholly Agile process were used?

Advantages of Agile:

Agile lets you adjust your website as you go along. If you need to change something, you can do it quickly without much hassle.
Agile helps you get parts of your website done and out to people faster and improving your website sooner.
Agile encourages everyone working on the website to talk and work together. Teamwork can help solve problems faster.
Agile makes sure your website is checked for mistakes regularly and makes people happy when they use it.

Challenges of Agile:

Agile needs clients to be involved and give feedback throughout the project.
With Agile, it's easy to keep adding new things to your website. This can make it hard to stay focused on what's most important.
Agile doesn't focus much on planning everything out at the beginning. Lack of planning can cause problems later on.
